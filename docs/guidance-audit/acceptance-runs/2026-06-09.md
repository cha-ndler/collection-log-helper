**Why the old FAIL no longer applies.** The 06-08 FAIL ("static dead-end — C1 unwired
`ACTOR_DEATH`") described step[3] declaring `ACTOR_DEATH` with neither `completionNpcId` nor
`completionNpcIds`, so no real NPC death could ever satisfy it and the chain froze. PR #781
converted step[3] to `MANUAL`. On a multi-item source the terminal "kill repeatedly for drops"
step is *meant* to stay live until a clog slot unlocks; the sequence force-completes via
`onItemObtained`, not via the step's own condition. This is identical in shape to the Shades of
Mort'ton terminal MANUAL the board already classifies as by-design (#729 control, 06-08).

**Engine receipt (unit tier, this run — `./gradlew test`, BUILD SUCCESSFUL):**

`GuidanceSequencer.onItemObtained(int)` completes the sequence when the obtained id is one of
the active source's target items, regardless of the current step's own (un-fireable) condition:

```java
// GuidanceSequencer.java
log.info("Target slot unlocked for {} (itemId={}) — completing guidance", ...);
fireSequenceComplete();
```

Covered green by `GuidanceSequencerTest`:

- `testObtainingTargetItemCompletesGuidanceRegardlessOfStepState` — a terminal step whose OWN
  condition has not fired is force-completed by `onItemObtained(targetItemId)`; asserts
  `wasTargetSlotUnlocked()` true, the sequence-complete callback fires, and guidance deactivates.
  This is precisely the Catacombs step[3] behaviour (dark totem pieces 19679/19681/19683 are the
  source's target items).
- `testObtainingNonTargetItemDoesNotCompleteGuidance` — the companion guard (a non-target obtain
  does NOT complete) — so the force-complete is scoped to the guided slot only.

**Corpus receipt (data tier, this run):** `GuidanceConfigInvariantsRegressionTest` PASS with the
`ACTOR_DEATH_MISSING_NPC_CEILING` ratchet decremented `5 -> 0` (the C1 batch, incl. Catacombs).
No source now declares an `ACTOR_DEATH` step with no completion-NPC id.

**Verdict: the old FAIL is RETIRED — Catacombs step[3] is a by-design terminal MANUAL** (data
fix merged, engine force-complete unit-verified). The end-to-end live drive (steps 0..2 via
`playerMoved`, step[3] HOLD on injection, force-complete via injected `itemObtained 19679`)
remains **PENDING** an operator-run sweep because the harness launcher is currently faulting
